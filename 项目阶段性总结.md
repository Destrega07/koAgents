# 项目阶段性总结

> KO E&D 赋能中心（可口可乐销售赋能中心）阶段性进度回顾（截至 2026-06-15）

## 一、项目目标与演进路线

本项目目标是将“单页多智能体聊天室”逐步演进为“销售数字化看板 + 多智能体异步流水线”的综合系统，用于支撑 KO（Key Account/大客户）在潜客开发与合作客户运营中的研究、洞察、提案与舆情/动作监测全流程。

当前阶段的核心路线：

- 从“对话式生成”升级为“结构化产物（卡片/报告）+ 可编辑 + 可追溯 + 可复用”
- 从“在线依赖”升级为“离线优先（IndexedDB）+ 可备份/恢复”
- 从“结果展示”升级为“证据-洞察-行动的关系联动 + 场景化决策支持”

## 二、总体架构与关键技术完成情况

### 1. 前端组织形态（已完成并稳定）

- 单页应用形态：Hash 路由 + 模块化渲染函数（如 renderXxx）
- 关键状态：集中 state 管理，refresh 驱动渲染刷新
- UI 风格：卡片墙/弹窗/固定顶部工具条/统计条等完成统一视觉语言

### 2. Coze 联调（已完成并多轮加固）

- 通过 `/api/chat` 代理对接 Coze SSE（流式）
- 请求体协议已对齐：`bot_id/user_id/stream/message` 必须位于第一层
- SSE 解析策略已收敛：严格只取 `role==="assistant" && type==="answer"` 的内容
- 超时与断流韧性增强：扩大超时阈值、AbortError 友好降级、finally 强制 refresh

### 3. 数据层（已完成并扩展）

- IndexedDB 存储作为系统核心数据底座
- 扩展了独立仓库 `facts_7c_cards`，用于 7C 卡片库（证据/洞察/行动假设）
- 备份/恢复能力已扩展覆盖新增仓库，采用完全覆盖策略

### C. 7C 事实与洞察库（facts7c）

#### 1) 7C 卡片库导入（已完成并增强鲁棒性）

- 批量上传 Markdown，按卡片标题切片
- 提取 ```yaml``` 并解析为结构化对象，按 card_type 分类为 evidence/insight/action_hypothesis
- 标题切片兼容：允许 `CAR_B01：...` 这类“ID 后接中文冒号/标题”

#### 2) 关系视图（已完成并修复关键联动缺陷）

- 三列联动：行动 → 洞察 → 证据（或洞察 → 证据）
- 修复：在洞察列使用下拉筛选后行动列消失的问题（改为保留列但不显示卡）
- 修复：仅选洞察卡时，洞察→证据无法联动的问题（补齐 supporting_evidence_refs / source_basis / support_chain 等字段识别）

#### 3) 主键与显示规范（已完成关键治理）

- 修复跨文件短号（如 E01）导致的主键覆盖风险
- 主键统一使用全局唯一 id/card_id（并作为关系引用）
- UI 展示统一使用全局唯一 id，确保不重复、可追溯

#### 4) Markdown 混合字段兼容（已完成）

- 对洞察卡：支持从 Markdown `### Insight`、`### KO_Implication` 回填结构化字段

- 解决“导入成功但卡片为空/机会线索缺失”的问题

## 四、阶段性成果（对业务的可见价值）

- 形成“潜客开发 → 价值主张 → 剧本提案”的可复用流水线
- 形成“合作客户运营 → 周报舆情 → 动作预测”的离线化监测能力
- 建立“证据-洞察-行动假设”的结构化知识库与关系联动视图
- 引入编辑、落库、导出、备份/恢复等“可交付”能力，支撑真实业务使用

## 五、当前已知风险与后续建议

- 历史数据兼容：部分能力依赖“重新导入”才能把新解析逻辑写入存量 HTML/卡片对象
- 报告/卡片格式多样：建议持续扩充“导入诊断信息”（切片数、跳过原因、缺字段统计）以加速定位
- 关系引用规范：持续推动数据源端统一字段命名（id/card_id、supporting_evidence_refs、source_basis 等）

下一阶段建议聚焦：

- 7C 卡片“导入诊断可视化”与“关系视图的可解释性”（为何关联/缺失）
- KA 周报的图表兜底策略继续扩展到更多报告模板（最小可用图表集合）
- 将关键链路沉淀为可配置策略（字段映射/解析器规则），降低后续维护成本
