# 公司信息录入与证据卡抽取规范

## 1. 文档目标

本规范用于指导将**公司侧原始信息**沉淀为可复用、可检索、可与客户分析报告联动的**公司证据卡**。目标有三点：

1. 先建立稳定的**公司证据层**，而不是过早固化公司洞察层。
2. 让产出的公司证据卡能够与客户侧证据卡、洞察卡、行动假设卡进行结构化碰撞。
3. 在字段天然不完整、来源格式复杂、人工资源有限的前提下，尽可能以**低人工依赖**方式跑通。

本规范先覆盖**快消公司通用场景**，再补充**KO（可口可乐公司/系统）个性化指引**。

---

## 2. 适用范围

适用于以下公司信息来源：

- 战略/年度回顾 PPT
- 生意复盘 Excel / CSV
- 访谈录音转写 / 访谈纪要 / work note
- 客户案例总结
- 供应链、渠道、产品、营销、价格、设备等内部材料
- 邮件纪要、项目总结、政策说明、培训材料

不适用于：

- 直接生成销售提案
- 未经校验即把内部口径升级为“高置信行动结论”
- 将愿景、口号、倡议性表述直接当成硬证据入库

---

## 3. 总体方法

## 3.1 核心判断

公司信息进入知识库时，优先录入**证据**，不优先录入**公司洞察**。

原因：

- 公司资料中“计划、观点、倡议、经验判断”比例高于客户年报类材料。
- 如果直接固化为洞察，极易把内部叙事误当成稳定能力。
- 行动假设卡需要的是“客户侧证据 × 客户侧洞察 × 公司侧证据 × 执行可达性”的交叉支撑，公司侧最稳的输入仍是证据。

因此建议采用以下链路：

**原始材料 → 结构切分 → 公司证据候选 → 公司证据卡 → 按客户任务检索调用 → 生成公司侧判断/行动假设支撑**

## 3.2 处理原则

1. **证据优先**：先抽已发生、已验证、已存在约束或能力的内容。
2. **来源分级**：同一结论若来自不同载体，优先级不同，不可混用。
3. **字段继承优先**：能从文件级、页级、表级、段级自动继承的字段，不要求人工逐条补写。
4. **缺失显式化**：字段空缺不可假装完整，必须标明“未知”“不适用”或“待补证”。
5. **兼容优先**：共享字段命名、选项池、状态值尽量与既有卡体系一致。
6. **低人工依赖优先**：优先靠预处理、模板化、规则路由、自动降级来跑通，不把流程设计成必须大量人工精修。

---

## 4. 公司证据卡在知识库中的定位

公司证据卡不是客户事实卡的替代品，而是行动假设生成时的另一组支撑材料。

它主要回答四类问题：

1. **公司有什么能力**：产品、包装、设备、研发、营销、数字化、供应链。
2. **公司有哪些约束**：最小起订量、价格带、履约限制、区域限制、法务/系统限制。
3. **公司过去做成过什么**：案例、试点、复制经验、成功边界。
4. **公司内部如何运作**：审批、owner、协同机制、决策节奏。

当客户侧出现某一机会或问题时，系统不应直接生成建议，而应先检索相关公司证据卡，判断：

- 是否存在匹配能力
- 是否存在能力边界
- 是否有相似案例
- 是否有商业或组织前提

## 5. 证据对象范围：优先录入的 6 类公司信息

以下 6 类信息应作为首批录入重点。

## 5.1 已验证的客户解决方案与成功案例

### 通用定义
已在真实客户、渠道、场景中执行过，并有结果反馈的项目、方案或合作模式。

### 应抽取内容
- 客户/渠道/场景对象
- 做了什么
- 在何时何地做
- 结果如何
- 成功条件
- 失败边界/限制条件

### 优先级
最高。它最能抬升行动假设的可行性与置信度。

### KO 个性化补充
优先录入：
- 联合营销/套餐/陈列/冰柜/即饮场景案例
- 包装规格切换案例（如不同容量、渠道定制规格）
- 餐饮连锁、便利、即时零售、校园、景区等典型场景案例
- 与大客户的联合创新或供货模式调整案例

## 5.2 产品 / 包装 / 设备 / 研发能力边界

### 通用定义
公司可提供什么，不可提供什么，什么可以改、什么不能改。

### 应抽取内容
- 产品线、SKU、包装规格
- 定制/改包/打样能力
- 研发与上新周期
- 设备适配要求
- 最小起订量 / 最小生产批次
- 法规或工艺限制

### KO 个性化补充
优先录入：
- 即饮、现调、餐饮专供、渠道专供等产品能力差异
- 包装规格、箱规、陈列设备、冷饮设备、糖度/风味定制边界
- 跨品牌组合可行性与限制
- 研发、试产、区域测试、系统推广的标准周期

## 5.3 供应链与履约能力

### 通用定义
产品能否按客户要求稳定送达，并以什么成本、什么频率、什么覆盖方式送达。

### 应抽取内容
- 覆盖网络
- 仓配模式
- 履约时效
- 断货率 / 满足率
- 小单高频支持能力
- 区域差异化履约能力
- 冷链/常温能力

### KO 个性化补充
优先录入：
- 装瓶厂网络覆盖与服务边界
- 直配、经销、联配模式差异
- 餐饮客户的补货频次、门槛、跨区域支持能力
- 加盟/连锁体系下的供货复杂度与协调成本

## 5.4 商业条件与盈利约束

### 通用定义
方案在商业上是否成立，包括价格、毛利、返利、费用支持、合同等条件。

### 应抽取内容
- 价格带 / 成本带
- 毛利下限
- 折扣/返利/促销支持机制
- 账期、回款约束
- 合同条款限制
- 不同渠道的商业模型差异

### KO 个性化补充
优先录入：
- 餐饮渠道与零售渠道的价格体系差异
- 不同包装规格对应的利润结构
- 低价引流产品的盈利边界
- 与客户联合促销时的费用承担规则

## 5.5 数字化与系统对接能力

### 通用定义
公司是否具备数据、系统、算法、接口、监控等能力支撑客户协同。

### 应抽取内容
- 是否有标准接口
- 已有系统能力
- 数据字段可开放范围
- 预测、补货、会员、库存、销售监控能力
- IT / 数据合规 / 法务前提

### KO 个性化补充
优先录入：
- 订货、补货、销量回传、冷柜/设备监测等能力
- 与大客户系统对接的历史案例
- 数据共享权限、频率、粒度要求
- 需要装瓶系统、区域团队、客户 IT 同时参与的门槛条件

## 5.6 组织资源与决策机制

### 通用定义
项目推进时谁能拍板、谁会阻挡、要拉哪些角色、周期多久。

### 应抽取内容
- 决策 owner
- 跨部门参与角色
- 立项门槛
- 审批流程
- 预算归属
- 推进周期

### KO 个性化补充
优先录入：
- 全国/区域/渠道/装瓶系统之间的职责边界
- Key Account、渠道、供应链、法务、财务、研发、市场等协同模式
- 哪些项目必须全国级批准，哪些可以区域试点
- 餐饮客户专项项目的常见推进路径与卡点

## 6. 证据来源分级

不同来源即使说的是同一件事，证据等级也不同。建议建立如下分级：

| 级别 | 来源类型 | 默认可信度 | 默认用途 |
|---|---|---:|---|
| A | 已发布经营数据、正式系统报表、正式项目复盘、签批文件 | 高 | 可直接作为 support |
| B | 正式 PPT、业务周报、部门总结、案例材料 | 中高 | 可作为 support / supplement |
| C | 访谈纪要、录音转写、会议纪要、邮件说明 | 中 | 多作为 supplement，必要时可支撑中低置信 |
| D | 个人判断、二手转述、无出处摘录 | 低 | 原则上不直接入正式证据卡 |

规则：

1. 单一 C 级来源，原则上**不能单独支撑高置信公司证据卡**。
2. 同一结论如果有 A/B/C 多源共振，应优先合并为一张卡，并抬升可信度。
3. D 级来源只进入待验证池，不直接进入正式库。

## 7. 不同媒介 / 载体的提取规则

## 7.1 PPT 提取规则

### 适合提取
- 战略方向
- 已发布能力
- 项目案例摘要
- 组织口径
- 关键 KPI 与阶段成果

### 主要风险
- 倡议性语言多
- “目标”“计划”“希望”容易被误当成事实
- 常缺少时间边界与适用边界

### 提取规则
1. 每条内容必须区分：**已发生 / 正在推进 / 未来目标 / 建议性表述**。
2. 只把“已发生”与“已明确存在的能力/约束”直接抽成正式证据候选。
3. 对“计划”“目标”“路线图”，可以抽取，但 `evidence_type` 必须标记为 `Plan` 或 `Intent`，默认不得作为高信心行动支撑。
4. 必须保留：文件名、页码、页标题、版本日期。
5. 一页有多个要点时要原子化拆分，不得整页糊成一张大卡。

### KO 个性化提示
- 将“全国战略口径”与“装瓶系统实际可执行能力”拆开抽取。
- 将“品牌愿景型表述”与“餐饮客户可调用能力”拆开抽取。

## 7.2 Excel / CSV 提取规则

### 适合提取
- 销量、收入、覆盖、价格、毛利、配送、试点结果、门店分布

### 主要风险
- 数字有口径但没有文字解释
- 表名、字段名、时间范围缺失
- 同一字段跨表含义不同

### 提取规则
1. 任何数字类证据都要绑定：表名、字段名、时间范围、统计口径。
2. 先抽取“指标事实”，后抽取“解释性备注”，两者不可混写。
3. 无时间范围的数字默认降级，不进高信心。
4. 无口径说明但数值明显重要的，可进入待校验池，不直接入正式库。
5. 对同一张表优先做批量字段继承，避免人工逐条补充。

### KO 个性化提示
- 对装瓶厂、渠道、客户、区域数据，必须附带地理层级和业务层级标签。
- 价格、销量、配送相关数据要区分“全国口径”“区域口径”“客户口径”。

## 7.3 访谈录音转写 / 访谈纪要提取规则

### 适合提取
- 隐性约束
- 真正的推进阻力
- 一线操作经验
- 组织偏好与内部机制

### 主要风险
- 主观判断比例高
- 说法带语境，脱离上下文容易失真
- 可能互相矛盾

### 提取规则
1. 必须记录：访谈日期、说话人角色、所属部门/地区、访谈主题。
2. 将访谈内容拆为：**事实陈述 / 经验判断 / 推测 / 建议** 四类。
3. 只有事实陈述与稳定经验判断可进入证据候选。
4. 单一访谈结论默认只作 `supplement`，除非有其他来源印证。
5. 访谈类证据卡默认置信上限不超过中，直到被 A/B 级材料补证。

### KO 个性化提示
- 对“总部这么想”和“区域/装瓶系统实际这么做”要分开记。
- 访谈中涉及客户敏感项目或尚未批准项目时，必须标记 `validation_status = provisional`。

## 7.4 Word / PDF 报告提取规则

### 适合提取
- 项目复盘
- 政策说明
- 年度总结
- 经营分析
- SOP / 流程文件

### 主要风险
- 一段里混合事实、评价、建议
- 章节结构复杂

### 提取规则
1. 先做章节切分，再做句级抽取。
2. 将“事实层”“解释层”“建议层”强拆。
3. 优先抽取带数字、带时间、带对象、带结果的句子。
4. 需要保留章节标题与页码/段号，便于追溯。

## 7.5 邮件 / IM / 会议纪要提取规则

### 适合提取
- 临时决策
- 项目 owner
- 时间节点
- 责任分工

### 主要风险
- 版本不稳定
- 表达简略
- 容易过期

### 提取规则
1. 必须保留发送日期、发送者角色、主题。
2. 将“确认事项”与“讨论事项”分开抽取。
3. 超过有效期的运营安排，不应继续作为当前支撑证据。
4. 邮件/IM 信息更多用于组织机制与推进约束，不宜单独支撑产品能力结论。

## 8. 公司证据卡推荐字段

## 8.1 必备字段

以下字段建议作为正式入库的最低门槛：

| 字段 | 说明 | 是否必备 |
|---|---|---|
| `id` | 唯一标识 | 是 |
| `card_type` | 固定为 `company_evidence` 或统一证据卡类型下的公司来源对象 | 是 |
| `title` | 单条原子化标题 | 是 |
| `entity` | 所属主体，如 KO / 区域 / BU / 装瓶系统 | 是 |
| `source_type` | PPT / Excel / Interview / Report / Email 等 | 是 |
| `evidence_type` | Capability / Constraint / Case / Performance / Process / Policy / Plan | 是 |
| `business_dimension` | 使用固定选项池 | 是 |
| `statement` | 规范化事实陈述 | 是 |
| `source_quote` | 原文摘录 | 是 |
| `source_locator` | 页码/表名/段落/时间戳 | 是 |
| `time_scope` | 时间范围 | 是 |
| `validation_status` | validated / provisional / pending / rejected | 是 |
| `confidence` | high / medium / low | 是 |

## 8.2 推荐字段

| 字段 | 说明 |
|---|---|
| `geo_scope` | 全国 / 大区 / 省区 / 城市 / 客户范围 |
| `channel_scope` | KA / GT / OOH / 餐饮 / 电商 / 即时零售等 |
| `customer_scope` | 适用客户类型 |
| `related_brands` | 关联品牌 |
| `related_skus` | 关联 SKU 或包装 |
| `constraint_tags` | 限制条件标签 |
| `success_conditions` | 成功成立所需条件 |
| `review_trace` | 校验过程摘要 |
| `claim_origin` | 来自原始事实 / 访谈 / 计划 / 系统数据 |

## 8.3 共享选项池要求

如与既有卡体系共享字段，应尽量沿用相同选项池。例如：

`business_dimension` 推荐与既有体系保持一致：

- `Channel`
- `Category`
- `Consumer`
- `Customer`
- `Competitor`
- `SupplyChain`
- `Pricing`
- `Brand`

如确需增加新维度，应先判断能否落回上述池，不得轻易自由发挥。

## 9. 字段空缺时如何处理

这是公司信息录入的核心难点。本节给出“少依赖人工”的标准处理方案。

## 9.1 字段分层：必须补、可继承、可缺省

### A. 必须补齐，否则不得正式入库
- `title`
- `entity`
- `source_type`
- `statement`
- `source_quote`
- `source_locator`
- `validation_status`
- `confidence`

### B. 优先自动继承，允许人工兜底
- `time_scope`
- `geo_scope`
- `business_dimension`
- `channel_scope`
- `claim_origin`

### C. 允许缺省，但必须显式标记
- `related_brands`
- `related_skus`
- `success_conditions`
- `constraint_tags`
- `customer_scope`

规则：

- 允许空值的字段必须写为 `unknown` / `not_applicable` / `not_stated`，不可静默留白。
- 留白会导致后续检索与判断误判，因此不推荐无标记空缺。

## 9.2 缺字段的自动补全顺序

当原文不具备完整字段时，按以下顺序自动补全：

### 第一层：文件级继承
从文件名、文档标题、版本号、所属部门、日期继承：
- `entity`
- `source_type`
- `time_scope`
- `geo_scope`

### 第二层：章节级继承
从章节标题、页标题、表标题继承：
- `business_dimension`
- `channel_scope`
- `customer_scope`
- `evidence_type`

### 第三层：内容级规则识别
通过关键词、模式匹配、列名识别补全：
- 若出现“起订量、MOQ、打样周期、产能限制”，优先判为 `Constraint`
- 若出现“试点、项目结果、复制、案例”，优先判为 `Case`
- 若出现“覆盖、配送、库存、补货、履约”，优先判为 `SupplyChain`
- 若出现“价格、毛利、返利、折扣”，优先判为 `Pricing`

### 第四层：缺失降级
若仍无法补齐：
- 将 `validation_status` 设为 `provisional` 或 `pending`
- 将 `confidence` 下调一档
- 允许入待校验池，不入正式主库

## 9.3 常见空缺字段的具体处理

### `entity` 缺失

#### 自动处理
按“文件归属 > 页眉/标题 > 章节主题 > 默认主实体”顺序推断。

#### 规则
- 若整份文件明确属于 KO 某部门/某区域，可批量继承。
- 若同一文件包含多个实体，必须按页或段落切分后再抽取。
- 无法确定时设为 `unknown_entity`，不得假定为 KO 总部。

### `time_scope` 缺失

#### 自动处理
从文件日期、页脚日期、表头周期、访谈日期中继承。

#### 规则
- 没有明确起止时间，但有文件发布日期时，可写 `as_of_YYYY-MM`。
- 对长期稳定能力，可写 `standing_capability`，但仍保留文件日期。
- 完全无时间信息则置信度降级。

### `business_dimension` 缺失

#### 自动处理
使用标题词、字段名、关键词路由到固定选项池。

#### 兜底规则
若同时命中多个维度：
- 以主语义为主维度
- 其余维度进入 `constraint_tags` 或检索标签

### `source_locator` 缺失

#### 自动处理
- PPT / PDF：页码
- Excel：sheet 名 + 行列区间
- 访谈：时间戳 + 说话人
- 邮件：日期 + 主题 + 段落

#### 规则
没有定位信息的内容原则上不入正式卡。

### `source_quote` 缺失

#### 自动处理
从原始载体抽取最小必要原文片段。

#### 规则
- 可做轻度清洗，但不可改写成摘要。
- 若只能得到转述而无原文，默认降为 `supplement` 候选。

### `confidence` 难判断

#### 自动计算建议
基于以下维度打分：
- 来源级别（A/B/C/D）
- 是否有明确定位
- 是否有时间边界
- 是否有多源印证
- 是否为已发生事实而非计划

示例：
- A 级 + 已发生 + 有定位 + 有时间 + 多源印证 → `high`
- B/C 级 + 有定位 + 单源 → `medium`
- C/D 级 + 缺时间或缺定位 + 带判断色彩 → `low`

## 10. 低人工依赖的运行方案

目标不是完全无人，而是**尽量把人工放在最后的少量把关，而不是前面的大量补字段**。

## 10.1 建议采用“四层自动化 + 一层抽检”

### 第 1 层：源文件预处理
自动完成：
- 文件类型识别
- 文档标题、版本、日期抽取
- 页码/表名/sheet 名抓取
- 访谈说话人和时间戳切分

### 第 2 层：结构切分
自动按：
- 页
- 章节
- 表
- 段落
- 访谈轮次
进行切分，并为每个区块挂载上级元数据。

### 第 3 层：候选抽取
自动区分：
- 已发生事实
- 能力陈述
- 约束陈述
- 计划/目标
- 建议/观点

只让前 3 类进入正式证据候选主通道。

### 第 4 层：字段继承与规则补全
自动从文件级、章节级、内容级补字段，并给出：
- 补全来源
- 不确定性标记
- 建议置信度

### 第 5 层：少量人工抽检
只抽检以下对象：
- 高价值高置信卡
- 缺关键字段但拟入正式库的卡
- 多实体混杂区块产出的卡
- 涉及客户敏感合作、系统对接、商业条款的卡

## 10.2 待校验池机制

为降低人工负担，不要把所有不完美卡片都拦在入口外。应设置两级池：

### 正式主库
要求：
- 必备字段齐全
- 有定位
- 有原文
- 时间清晰
- 非纯计划/纯建议

### 待校验池
适用于：
- 来源较弱
- 定位不完整
- 时间边界不清
- 仅能作补充支撑

好处：
- 不会因为字段不全而完全丢失信息
- 又不会污染正式主库
- 后续一旦拿到补证，可自动升级

## 10.3 自动降级规则

当以下情况出现时，系统应自动降级而非硬凑：

| 情况 | 自动处理 |
|---|---|
| 无原文摘录 | 降为待校验池 |
| 无来源定位 | 不入正式库 |
| 无时间边界 | 置信度降一档 |
| 仅为未来目标 | `evidence_type = Plan`，不得高置信 |
| 仅有单一访谈支撑 | 最高中低置信 |
| 多字段冲突 | 不合并，拆成多张候选卡 |

## 11. 推荐流程

## Step 1：源文件登记与元数据抽取
输出：文件级元数据表

至少抽取：
- 文件名
- 来源部门/团队
- 日期/版本
- 文件类型
- 主实体
- 保密级别

## Step 2：结构切分
输出：区块清单

切分维度：
- 章节
- 页
- 表
- 段
- 访谈轮次

## Step 3：区块分类
输出：区块类型标签

区分：
- 数据事实块
- 能力说明块
- 约束说明块
- 案例说明块
- 计划/愿景块
- 观点/建议块

## Step 4：证据候选抽取
输出：公司证据候选

只优先保留：
- 已发生事实
- 已存在能力
- 已存在约束
- 已验证案例

## Step 5：字段继承与补全
输出：标准化证据候选

补全过程需记录：
- 哪些字段来自原文
- 哪些字段来自继承
- 哪些字段来自规则推断

## Step 6：置信度与状态判定
输出：正式证据卡 / 待校验卡

## Step 7：入库与索引
输出：可检索公司证据卡库

建议索引字段：
- `business_dimension`
- `entity`
- `evidence_type`
- `geo_scope`
- `channel_scope`
- `related_brands`
- `constraint_tags`

## 12. 与客户侧卡片联动的调用规则

公司证据卡不应全量扫描后再生成行动假设。建议采用“先检索，再碰撞”。

## 12.1 调用顺序

1. 读取客户侧证据卡与洞察卡
2. 提取检索键：
   - 业务维度
   - 场景词
   - 包装/设备词
   - 价格/供应链/数字化关键词
3. 只召回最相关公司证据卡
4. 检查是否形成：
   - 客户问题存在
   - KO 能力存在
   - KO 约束已知
   - 有相似案例或至少有执行路径
5. 再决定是否生成行动假设卡

## 12.2 行动假设置信度提升条件

从低/中提升到中/高，通常至少需要满足：

1. 客户侧证据充分
2. 客户侧洞察成立
3. 公司侧存在明确能力或成功案例
4. 商业条件未明显冲突
5. 组织路径可达

只满足前两项时，通常只应得到低信心行动假设。

## 13. KO 公司信息录入的优先顺序

建议 KO 首批录入按以下顺序推进：

### 优先级 1：可直接支撑行动可行性的材料
- 连锁餐饮客户成功案例
- 包装规格与设备适配说明
- 供应链/补货/配送能力说明
- 费用与商业条款边界

### 优先级 2：可判断组织推进难度的材料
- 项目审批与 owner 机制
- 全国/区域/装瓶职责边界
- 系统对接前提与法务要求

### 优先级 3：可作为补充语境的材料
- 战略方向 PPT
- 访谈经验
- 内部趋势判断

注意：

- 不建议优先大量录入纯战略愿景材料。
- 不建议优先录入未经验证的“未来客户打法”。
- 不建议把内部提案文档直接卡化为能力证据。

## 14. KO 个性化字段与标签建议

在不破坏通用兼容性的前提下，可为 KO 增加若干检索标签，但这些标签应作为扩展标签，不替代共享字段。

建议扩展标签：

- `system_scope`: `KO_corporate` / `bottler` / `joint`
- `serve_model`: `direct` / `distributor` / `hybrid`
- `solution_type`: `package` / `equipment` / `promotion` / `menu_bundle` / `digital_interface` / `supply_chain`
- `customer_archetype`: `QSR` / `casual_dining` / `convenience` / `instant_retail` / `campus` / `tourist`

这些标签主要用于提升后续检索效率。

## 15. 质量门槛

## 15.1 正式入库最低标准

一张公司证据卡至少应满足：

- 有明确主语（谁的能力/约束/案例）
- 有明确陈述（发生了什么/能做什么/不能做什么）
- 有原文摘录
- 有定位信息
- 有时间边界或生效边界
- 非纯建议、非纯愿景、非纯推测

## 15.2 应拒绝入库的情况

- 完全无出处
- 只有转述没有原文
- 建议性语言占主体
- 未来目标被当成已发生事实
- 多个事实、多个维度、多个主体混在一张卡中

---

## 16. 常见错误与纠偏

| 常见错误 | 错误表现 | 纠偏方式 |
|---|---|---|
| 把战略愿景当能力 | “未来要打造…” 被写成已有能力 | 标记为 `Plan`，不得高置信 |
| 把访谈观点当硬事实 | “我觉得客户会接受” 被直接入卡 | 降为 supplement 或待校验 |
| 把案例摘要写成建议 | “因此应复制到A客户” | 删除建议层，只保留案例事实 |
| 缺时间边界 | 有结论，无时段 | 继承文件日期或降级 |
| 缺实体 | 不知是总部还是区域 | 用元数据继承；无法确定则标 `unknown_entity` |
| 一卡多义 | 同时写能力、约束、案例 | 拆卡 |

## 17. 结论

对快消公司而言，最稳妥的公司信息沉淀方法，不是先造公司洞察，而是先建设**公司证据库**。对 KO 而言，这一步尤其重要，因为很多行动假设之所以长期停留在低/中置信，并不是客户侧问题不成立，而是公司侧能力、边界、案例、组织路径没有被结构化录入。

因此推荐执行顺序为：

1. 先录入 6 类高价值公司信息；
2. 先做公司证据卡，不急于做公司洞察卡；
3. 用字段继承、规则补全、待校验池、自动降级机制，降低对人工补字段的依赖；
4. 在客户任务场景中按需检索公司证据卡，再与客户侧证据/洞察碰撞生成更高质量的行动假设。

这套规范的目标不是让每条原始信息都完美，而是让系统在不完美输入下，依然能稳定地产出**可信、可追溯、可复用**的公司侧证据对象。
