我们需要彻底重构系统的【提案剧本】（`renderProposal`）模块。该模块需要无缝接收由【价值主张创造】上游灌注的联邦上下文、联动调用 `<万能剧本提案高手>`（Bot ID: 7553662475095867442）、利用打字机效果流式渲染、并支持手工二次改动与文本导出。

# 详细重构需求说明

## 1. 跨组件五源数据联邦打包 (重构【万能剧本提案】按钮触发器)
重构【价值主张创造】（`renderValueProp`）子模块底部的 `{万能剧本提案}` 按钮点击事件，当满足点亮条件且用户点击时，前端需执行以下“深度多源数据包组装”：
- **数据源 A（已购选的方案与价值主张）**：遍历并收集用户在【价值主张创造】页面打勾选中的所有方案、对应的价值主张及其需求分组头。
- **数据源 B（独特性分析与合成）**：获取当前页面的 “方案独特性分析” 与 “价值主张合成” 内容区块。
- **数据源 C（饮料角色）**：从 IndexedDB 或上游缓存中读取【需求验证与洞察】阶段锁定的 “饮料角色判断”。
- **数据源 D（全量客户档案）**：通过 `state.selectedCustomerId` 从 IndexedDB 中提取该客户的**全部档案字段**（如店数、餐饮类型等）。
- **打包分发**：将上述数据统一压缩为一个名为 `proposal_federal_context` 的 JSON 对象。将其作为 `message` 参数包，调用 `/api/chat` 流式代理接口，目标指向 `bot_id: "7553662475095867442"`。
- ** Hash 锚点跳转**：发起请求的同时，更新 Hash 指针，系统自动平滑跳转至【提案剧本】（`proposal`）子模块页面。

## 2. 统一流式打字机渲染与专业级排版节奏 (重构 `renderProposal`)
- **防止双份与假死**：借鉴之前模块的成熟去重清洗经验，在 `extractCozeAssistantText` 阶段严格锁定 `type === "answer"`。一旦检测到流式数据，首先 `clearNode` 清空旧面板，开启 Loading 提示条：*“可口可乐万能剧本提案高手正在为您全力转译商业逻辑、生成实战话术剧本中，请稍候...”*。
- **像素级样式对齐**：提案剧本页面卡片容器左右 Padding 严格锁定为 `24px`，确保卓越的页面呼吸感。
- **流式字数承接与 Markdown 渲染**：支持一边接收流式字符，一边通过 `markdownToSafeHtml` 将提案剧本转换为美观、错落有致的 HTML 展现，确保剧本中的 `【适用场景】`、`【沟通策略】` 高亮易读，关键字加粗。

## 3. 实现手工二次改动与持久化
- **可编辑沙盒**：整个生成的提案剧本报告区域，必须包裹在一个支持二次编辑的容器中。用户既可以直接在各行段落上双击进行手工录入、编辑与文字校准。
- **自动落库**：用户二次编辑修改时，通过防抖（Debounce）或离焦（`blur`）事件，调用 `setModuleData(cid, "proposal", { content: 最新修改后的文本 })` 实时持久化回填入 IndexedDB。

## 4. 系统级导出组件升级 (TXT 与 调试沙盒)
在页面显著位置常驻两个样式精美的业务按钮：
- **`{导出txt文件}` 按钮**：一键读取当前最新编辑保存的文本，打包生成标准的 `.txt` 文本文件并触发浏览器自动下载，命名规范为 `提案剧本_[客户名]_[YYYYMMDD].txt`。
- **`{导出PNG长图}` 按钮**：暂时留好高亮样式与 `onclick` 事件接口（绑定 toast("导出长图组件正在封装中...") 提示），为第三阶段长图渲染预留。
- **底部调试沙盒**：最下方保留 `<details>` 标签包裹的“Coze端原始对话分析内容折叠区（调试）”，展现清洗后的 Markdown内容。
