# 角色与目标
你是一位在流式传输协议（SSE）和分布式多智能体数据对齐领域拥有深厚经验的顶级全栈工程师。
我们在 `app.js` 的【提案剧本】（`renderProposal`）模块遇到了严重的底层设计缺陷：一是流式读取器与后端代理协议未完全兼容，导致请求被断锁在本地，未成功唤醒扣子端；二是前端编写了过度妥协的回填逻辑，将发送给 AI 的原始输入上下文直接拼凑伪装成了 AI 的报告内容。
本轮任务：彻底打通底层流通道、斩断前端伪造回填逻辑，确保呈现真正的扣子端实战提案。

# 核心缺陷修复与开发规格

## 1. 修复流式网络请求器通道 (重构 app.js 中的 `streamCozeAnswerText`)
当前扣子端没有接收记录，根本原因在于 `streamCozeAnswerText` 在通过 `fetch` 启动请求时，尽管携带了 `Accept: text/event-stream`，但其内部对长链接流的解析触发机制和异常挂起没有与后端服务完全跑通。
- **对齐代理转换**：请检查并修复 `streamCozeAnswerText` 的网络底座。必须确保发给后端的参数中 `stream: true` 处于纯粹干净的最外层。
- **增强鲁棒性监听**：引入针对 `onDelta` 的防死锁保障。在 `fetch` 接收响应的刹那，如果超过 3 秒内未收到任何合法的 `data:` 协议头部，或者接收到的 HTTP 状态码为非 200，必须立刻触发 `reject` 或 `throw new Error("后端流转接管道建立失败，系统自动尝试降级同步握手")`。

## 2. 斩断前端伪造逻辑，严禁“自制假报告”覆盖 (修复 app.js 内的 `runProposalPipeline`)
- **根除自欺欺人缺陷**：在 `runProposalPipeline` 中，彻底删除或注销任何形如“如果 `finalText` 为空，则把 `proposal_federal_context` 的原始入参用 JS 串联拼凑成 Markdown 写入落库”的兜底伪逻辑。
- **数据源唯一原则**：IndexedDB 的 `proposal` 模块内容必须 100% 且唯一来源于 `<万能剧本提案高手>` 的 `answer` 节点回复。如果 AI 未能返回，页面必须报错或保持 Loading 占位状态，绝不允许为了防止白屏而自己拼凑文本，这会彻底掩盖真实网络未通的 Bug。

## 3. 动态维护可视化加载状态机
- **清除旧数据痕迹**：当销售人员在【价值主张创造】模块点击 `{万能剧本提案}` 的刹那，系统在执行 `runProposalPipeline` 前，必须隐式将本地数据字典中该客户的 `proposal` 缓存字段彻底清空，或将其设为 `"生成中…"`。
- **Loading 防御**：进入【提案剧本】模块页面时，由于老数据已被清空，如果真理请求还在异步传输中，系统必须呈现你开发的精美红色 Loading 动效，文案严格固化为：*“可口可乐万能剧本提案高手正在为您全力转译商业逻辑、生成实战话术剧本中，请稍候...”*。只有当 Coze 端真正的流式文本完全清洗完毕、落库成功后，再整体触发 `refresh()` 展开大段美观的话术展示，杜绝出现跳转过去直接显示入参的乱象。

## 4. 样式与标签高亮对齐 (检查并优化)
- 确保在 Markdown 解析并渲染进入 `.proposal-viewer` 容器后，原先利用 `replace` 替换的 `【适用场景】`、`【沟通策略】`、`【策略】`、`【话术示例】` 等高亮标签样式在 PC 桌面端大屏幕下具备正常的 24px 页边距，字重为 `800`，使销售能够一目了然看懂话术层级。
