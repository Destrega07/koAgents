你是一位精通文本状态机深度清洗与 Markdown 语法树分片渲染的高级全栈工程师。
我们在 `app.js` 中的 `sliceValuePropMarkdown` 正则提取逻辑存在严重回溯漏洞，导致“针对需求：...”行被错误判定为上一个方案的未完结追加文本，从而混入了 Slot 卡片内部并破坏了后续的流式分组。

# 核心漏洞修复与开发规格

## 1. 重写文本流状态机拦截边界 (彻底修复 app.js 内的 `sliceValuePropMarkdown`)
请重构 `sliceValuePropMarkdown` 函数中关于 `plansBlock` 逐行解析的 `for (const rawLine of lines)` 循环状态机。必须建立“高优先级前瞻阻断”防御机制：
- **前置阻断校验**：在循环每一行时，处理 `if (cur && tt)` 向后追加逻辑前，**必须首先**进行以下正则判定：

```javascript
  // 建立鲁棒性更高的需求标题行判定（包容开头可能存在的 Markdown 符号、加粗符号或空格）
  const needHeaderRe = /^\s*(?:[-*]\s*)?(?:\*\*)?针对需求\s*[:：].*$/;
```
- 强制斩断状态机：只要检测到当前行满足 needHeaderRe，必须立即强制执行 pushCur() 将上一个方案卡片封包归仓，并提取其中的需求文本更新为最新的全局 curGroup 变量：

```JavaScript
  if (needHeaderRe.test(tt)) {
      pushCur(); // 立即封包上一个方案，断开追加链路
      // 提取针对需求后面的核心描述文本作为 group 头
      const cleanGroup = tt.replace(/^\s*(?:[-*]\s*)?(?:\*\*)?/, "").trim();
      curGroup = cleanGroup;
      continue; // 强行跳过当前行，绝对不允许其进入下方的追加逻辑
  }
```

## 2. 完善方案卡片正向匹配规则 (修复 app.js)
方案正向匹配修改：由于 Coze 输出的方案可能为“方案一：”、“方案1：”或带有 Markdown 加粗，请将 planRe 泛化拓宽：

```JavaScript
  const planRe = /^\s*(?:[-*]\s*)?(?:strong)?方案\s*[一二三四五六七八九十0-9]+\s*[：:]\s*([\s\S]*)$/i;
```

确保当且仅当一行确凿满足方案特征时，才开启新卡片 cur = { group: curGroup, ... } 的创建。

## 3. 分组组件隔离渲染 (检查并优化 renderValuePropPlanSlots)
确保 renderValuePropPlanSlots 严格按照清洗后干干净净的 slot.group（形式如：针对需求：外卖适配支持：...）进行聚集分组。

DOM渲染对齐：每个需求分组渲染时，必须先单独向容器挂载一个独立的 .slot-group-line 块级节点呈现需求声明，下方再挂载该组专属的、绝不夹杂任何需求文本纯净方案 的 .slot-card 复选 Slot 卡片矩阵。

## 交付标准
