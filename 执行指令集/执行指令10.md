# 核心缺陷修复与开发规格

## 1. 修复网络层流式死锁 Bug (彻底重写 app5.js 内的 `runValuePropPipeline`)
目前 `runValuePropPipeline` 在向后端发送请求时，由于网络参数包装层级不正确，导致后端代理未识别出 `stream` 协议，使前端流式读取器 `getReader()` 陷入无限死锁挂起状态。
- **协议对齐重构**：请重构 `runValuePropPipeline` 内部调用 `/api/chat` 的参数包装。必须确保 `bot_id`、`user_id`、`stream: true` 处于 POST 请求体的**最外层（First-level fields）**。
- **消息体注入**：将由上游模块联邦组装的 `pipeline_context` 结构化对象转为 JSON 字符串，作为最外层的 `message` 字段分发：
```javascript
  const reqBody = {
    bot_id: "7552194182392512566",
    user_id: getOrCreateUserId(),
    stream: true, // 确保处于第一层级，激活后端的 SSE 转发代理
    message: JSON.stringify(pipeline_context)
  };
```

## 2. 升级清洗过滤逻辑，确保不重复解析报告
- 优化 extractLikelyValuePropMarkdownFromSseData 函数。严禁使用宽泛的 walk 机制深度遍历 verbose 节点的嵌套 JSON。

- 单一数据源提取：必须严格判定 Coze API 下发的消息包属性。优先且仅提取 role === 'assistant' 且 type === 'answer' 消息节点的 content 进行累加（delta）或覆盖（completed）。彻底屏蔽 type === 'verbose' 带来的二次重叠污染，一举根除出现两份重复报告的隐患。

## 3. 补齐【价值主张创造】子模块的可视化状态机提示
- 流式 Loading 提示：在智能体流式计算期间（即 state.ui.valuePropLoading.has(cid) 为真时），前端页面必须常驻渲染出带有可口可乐红动效的 ai-loading 状态条。

- 文案规范：提示词统一呈现为：“可口可乐价值主张创造参谋正在深度匹配资源、推演差异化方案中，请稍候...”，避免由于白屏白等待导致用户误以为系统死机。

## 4. 优化 table 样式展现与常驻原始内容
- 呼吸感对齐：为价值主张页的所有专属组件 .valueprop-editor 与生成的原生 <table> 补充 24px 的左右对边距（Padding），维持模块统一的高级感视觉规范。 

- 常驻原始内容调试区：在页面最下方的 <details> 调试沙盒区中，将系统从未解析清洗前收集到的流式原始原始文本（rawStream）一并进行原样呈现，以便我们进行生产环境的数据核对。
